## New critical problems introduced

### NP1 — `mission-control.json` replaces `platform.conf` — direct README violation

The plan states:
> "Single Source of Truth: `/mnt/${TENANT_ID}/config/mission-control.json`"
> "Script 2: Read mission-control.json (not platform.conf)"

The README states explicitly:
> "`platform.conf` — the single source of truth for the entire platform. Every key generated by script 1. Read by scripts 2, 3, and 0."

The README does not mention `mission-control.json` anywhere. The README's `platform.conf` reference section lists 80+ keys that scripts 2, 3, and 0 read directly by sourcing the file with `source platform.conf`. Replacing this with a JSON file requires rewriting the consumption pattern in every script, breaks the `source` idiom, and introduces `jq` as a mandatory runtime dependency for every variable access.

**This is not a minor deviation.** It changes the fundamental data flow the README was designed around.

**Fix required:** Retain `platform.conf` as primary source of truth. `mission-control.json` may exist as a **derived artifact** generated from `platform.conf` for use by tooling that prefers JSON — but scripts 2, 3, and 0 must still `source platform.conf`.

### NP2 — `generate_compose()` builds YAML by concatenating `jq` output in a loop — guaranteed to produce malformed YAML

The plan's `generate_compose()` function:
```bash
jq -r '.services | to_entries[] | @base64' "${mc_file}" | while read -r service; do
    generate_service_block "${name}" "${config}"
done
```

`generate_service_block` returns shell stdout that gets appended to the compose file inside a pipe subshell. Problems:

1. The `while read` loop runs in a subshell — any variables set inside are lost
2. Each `generate_service_block` call appends unindented YAML fragments — indentation in docker-compose YAML is structural, not cosmetic
3. No mechanism ensures `depends_on` references are only written for services that are actually enabled — a disabled service referenced in `depends_on` causes `docker compose up` to fail
4. `version: '3.8'` is deprecated in Compose Spec and produces a warning that can break `--quiet` validation

**Fix required:** Generate the entire compose file in a single heredoc with explicit conditional blocks per service, controlled by the `*_ENABLED` flags from `platform.conf`. This is what the original script 2 did correctly. The loop approach is fragile for YAML generation.

### NP3 — Zero hardcoding scan will fire on legitimate values

The plan's validation step:
```bash
if grep -rE "localhost|127\.0\.0\.1|http://|https://|sk-|CHANGEME" \
  "${BASE_DIR}/config/"*.yml "${BASE_DIR}/config/"*.yaml; then
    fail "Hardcoded values detected in generated configs"
fi
```

Problems:
1. `http://` and `https://` — the Caddyfile reverse proxy blocks use `http://` for upstream service references (e.g., `reverse_proxy http://open-webui:3000`). This will always trigger a false positive and block deployment.
2. `localhost` — Redis and Postgres health checks use `localhost` internally within the container (e.g., `redis-cli -h localhost ping`). False positive.
3. `sk-` — if a user legitimately configures an OpenAI API key starting with `sk-` in their LiteLLM config (which is the standard OpenAI key format), this blocks deployment.

**Fix required:** Remove `http://`, `https://`, `localhost`, and `sk-` from the scan. Only scan for `CHANGEME`, `TODO`, `FIXME`, `xxxx`, and empty required fields.

### NP4 — Caddyfile validation pulls `caddy:2-alpine` at validation time — fails in air-gapped or rate-limited environments

```bash
docker run --rm -v "${BASE_DIR}/config/caddy/Caddyfile:/etc/caddy/Caddyfile:ro" \
  caddy:2-alpine caddy validate --config /etc/caddy/Caddyfile
```

This requires an internet pull at script 1 validation time, before any services are deployed. If the host has not yet pulled the caddy image, this fails. If Docker Hub rate limits apply, this fails. Script 1 runs before script 2 — caddy is not yet pulled.

**Fix required:** Either (a) perform Caddyfile validation inside script 2 after images are pulled, or (b) use a static Caddyfile linter that does not require Docker (e.g., a regex check for unmatched braces and required site blocks), or (c) check if the caddy image is already present locally and skip the docker validation if not, with a warning.

### NP5 — `apply_stack_preset()` in `full` preset hardcodes all vector DBs as enabled

```bash
"full")
    QDRANT_ENABLED=true
    WEAVIATE_ENABLED=true
    CHROMA_ENABLED=true
    MILVUS_ENABLED=true
```

Milvus in standalone mode requires `etcd` and `minio` containers in addition to the `milvus` container itself. The plan does not generate compose blocks for etcd and minio. Enabling Milvus without these will produce a container that starts and immediately exits. This was Conflict C7 in my original review — it is **not resolved** in the corrected plan.

**Fix required:** Either (a) implement full three-container Milvus compose block (etcd + minio + milvus), or (b) remove Milvus from the `full` preset until it is fully implemented, with a clear TODO comment.

### NP6 — `generate_litellm_config()` uses `envsubst` on a heredoc that already has variables expanded

```bash
cat > "${config_dir}/config.yaml" << EOF
  master_key: "${LITELLM_MASTER_KEY}"
EOF
envsubst < "${config_dir}/config.yaml" > "${config_dir}/config.yaml.tmp"
mv "${config_dir}/config.yaml.tmp" "${config_dir}/config.yaml"
```

The heredoc with double-quoted `EOF` already expands shell variables at write time. Running `envsubst` afterward on the already-expanded file is a no-op at best and destructive at worst — if any expanded secret value contains a `$` character (which is possible in generated passwords), `envsubst` will attempt to expand it as an environment variable, corrupting the value.

**Fix required:** Choose one approach: either use the heredoc with variable expansion (remove `envsubst`), or use a literal heredoc (`<< 'EOF'`) with `envsubst` for expansion. Do not use both.

---

### NP7 — `framework_validate()` requires `yq` but `yq` is not in the README's system package installation list

The plan adds `yq` as a required binary in pre-flight checks. The README's script 1 package installation section lists:
```
docker, docker-compose-plugin, curl, jq, openssl, rclone, cron
```

`yq` is not listed. If `framework_validate()` runs before `yq` is installed and `yq` is absent from the system, the script fails before it can install dependencies.

**Fix required:** Add `yq` to the package installation step in script 1, before `framework_validate()` runs. Or replace the `yq` YAML validation with `docker compose config --quiet` (which validates YAML syntax as a side effect of parsing the compose file) and eliminate the `yq` dependency entirely.

## Two architectural decisions requiring explicit decision before implementation

### AD1 — `platform.conf` vs `mission-control.json` as primary source of truth

This must be decided now, not during implementation:

**Option A (README-compliant):** `platform.conf` is primary. Scripts source it directly. `mission-control.json` is a derived JSON view generated for tooling use only.

**Option B (plan's proposal):** `mission-control.json` is primary. Scripts use `jq` to read all variables. `platform.conf` is deprecated or removed.

Option A requires no changes to the README or the consumption pattern in scripts 2, 3, 0. Option B requires rewriting all variable access patterns and adding `jq` expressions for every variable read — approximately 200+ variable accesses across three scripts.

**Recommendation: Option A.** The plan should be corrected to reflect this.

### AD2 — Config generation location: script 1 or script 2

The corrected plan moves all config generation (LiteLLM, Bifrost, Caddyfile, docker-compose) into script 1. The README assigns:
- Script 1: input collection, `platform.conf` generation, directory structure
- Script 2: read `platform.conf`, generate all configs, deploy containers

If config generation moves to script 1, script 2 becomes a thin wrapper around `docker compose up`. This breaks the README's explicit responsibility boundary and means re-running script 1 (e.g., to add a service) requires re-running script 2 to re-deploy — which is what the README design already requires, but now script 1 is doing script 2's job.

**Recommendation:** Keep config generation in script 2 as the README specifies. Script 1 generates only `platform.conf` and directory structure. Script 2 reads `platform.conf` and generates all derived config files before deploying.
