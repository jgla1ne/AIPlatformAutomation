### C1 — Script 2 sources itself recursively via `append_*` functions

**File:** `scripts/2-deploy-services.sh`  
**Problem:** Script 4 does `exec bash scripts/2-deploy-services.sh` after updating `.env`. If script 2 calls any function that re-sources script 2, you get infinite recursion. More critically, if script 2 defines functions like `append_postgres()` etc. but calls them in `main()` which is invoked at the bottom, and script 4 sources script 2 to get those functions — the `main "$@"` at the bottom of script 2 fires immediately on source. This is a shell script anti-pattern that will cause double-execution.

**Fix:** Add a guard at the bottom of script 2:
```bash
# Bottom of 2-deploy-services.sh — replace bare main call with:
if [[ "${BASH_SOURCE[0]}" == "${0}" ]]; then
    main "$@"
fi
```

### C2 — `generate_compose()` heredoc quoting will break variable expansion

**File:** `scripts/2-deploy-services.sh`  
**Problem:** Every `compose_append << EOF` (unquoted EOF) expands shell variables at write-time. This is correct for most values. But variables like `${POSTGRES_PASSWORD}` that contain special characters (generated by `openssl rand`) will break the YAML if they contain `$`, `\`, backticks, or `!`. A password like `aB$3\nX!` written into a heredoc becomes corrupted YAML.

**Fix:** All passwords must be base64-encoded or constrained to alphanumeric-only at generation time in script 1:
```bash
# In script 1 — replace openssl rand -base64 with:
generate_password() {
    # Alphanumeric only — safe in heredocs, YAML, env vars
    tr -dc 'A-Za-z0-9' < /dev/urandom | head -c 32
}
ADMIN_PASSWORD=$(generate_password)
POSTGRES_PASSWORD=$(generate_password)
REDIS_PASSWORD=$(generate_password)
MINIO_ROOT_PASSWORD=$(generate_password)
LITELLM_MASTER_KEY="sk-$(generate_password)"
FLOWISE_PASSWORD=$(generate_password)
GRAFANA_PASSWORD=$(generate_password)
N8N_PASSWORD=$(generate_password)
```

### C3 — Docker network teardown in script 2 will fail if containers are still attached

**File:** `scripts/2-deploy-services.sh` — `teardown_existing()` function  
**Problem:** `docker network rm` fails with `"has active endpoints"` if any container is still attached. The teardown must stop containers **before** removing the network, and must handle the case where compose file does not yet exist (first run).

**Fix:**
```bash
teardown_existing() {
    log "INFO" "Tearing down existing deployment..."

    local compose_file="${DATA_ROOT}/docker-compose.yml"

    # If prior compose file exists, use it to bring down cleanly
    if [ -f "${compose_file}" ]; then
        docker compose \
            --project-name "${COMPOSE_PROJECT_NAME}" \
            -f "${compose_file}" \
            down --remove-orphans --timeout 30 2>/dev/null || true
    fi

    # Belt-and-suspenders: force-remove any lingering containers
    docker ps -aq --filter "name=${COMPOSE_PROJECT_NAME}" | \
        xargs -r docker rm -f 2>/dev/null || true

    # NOW safe to remove network
    docker network ls --filter "name=${COMPOSE_PROJECT_NAME}" -q | \
        xargs -r docker network rm 2>/dev/null || true

    # Prune dangling networks
    docker network prune -f 2>/dev/null || true

    log "SUCCESS" "Teardown complete"
}
```

### C4 — `COMPOSE_PROJECT_NAME` and `DOCKER_NETWORK` not exported to docker compose environment

**File:** `scripts/2-deploy-services.sh`  
**Problem:** When script 2 calls `docker compose --project-name ${COMPOSE_PROJECT_NAME} -f ${compose_file} up`, the generated `docker-compose.yml` references `${DOCKER_NETWORK}` inside the networks section. But at runtime, docker compose reads the compose file and tries to expand `${DOCKER_NETWORK}` — it looks in the environment **at docker compose invocation time**, not at script generation time. If the heredoc already substituted the real value into the file at generation time, this is fine. But if any `${VAR}` survived into the written file literally, docker compose will look for it in `.env` at the compose file location — which is `/mnt/data/u1001/docker-compose.yml` but the `.env` is also at `/mnt/data/u1001/.env`. Docker compose auto-loads `.env` from the **same directory as the compose file** — so this actually works, **only if** both files are in the same directory.

**Verify this is true:** Confirm in script 2:
```bash
# Both must be in same directory
COMPOSE_FILE="${DATA_ROOT}/docker-compose.yml"
ENV_FILE="${DATA_ROOT}/.env"
# DATA_ROOT is same for both ✅ — this is correct
# But docker compose must be invoked with --env-file explicitly to be safe:
docker compose \
    --project-name "${COMPOSE_PROJECT_NAME}" \
    --env-file "${ENV_FILE}" \
    -f "${COMPOSE_FILE}" \
    up -d ...
```

Add `--env-file "${ENV_FILE}"` to **every** `docker compose` call in script 2.

### H6 — Script 1 GPU detection method is unreliable on AWS

**File:** `scripts/1-setup-system.sh`  
**Problem:** `nvidia-smi` may not be in PATH even when GPU is present if NVIDIA container toolkit isn't installed yet. Script 1 should detect GPU differently:

```bash
detect_gpu() {
    # Method 1: nvidia-smi
    if command -v nvidia-smi &>/dev/null && nvidia-smi &>/dev/null; then
        GPU_TYPE="nvidia"
        GPU_COUNT=$(nvidia-smi --query-gpu=name --format=csv,noheader | wc -l)
        log "INFO" "NVIDIA GPU detected: ${GPU_COUNT} GPU(s)"
        return
    fi

    # Method 2: lspci
    if command -v lspci &>/dev/null; then
        if lspci 2>/dev/null | grep -qi "nvidia"; then
            GPU_TYPE="nvidia"
            log "WARN" "NVIDIA GPU found via lspci but nvidia-smi unavailable"
            log "WARN" "Install: sudo apt install nvidia-container-toolkit"
        elif lspci 2>/dev/null | grep -qi "amd.*display\|radeon"; then
            GPU_TYPE="amd"
        fi
        return
    fi

    # Method 3: /proc/driver/nvidia
    if [ -d "/proc/driver/nvidia" ]; then
        GPU_TYPE="nvidia"
        return
    fi

    GPU_TYPE="cpu"
    log "INFO" "No GPU detected — using CPU mode"
}
```
